## 6. 기획·코딩 작업자를 위한 핵심 팁

### Thinking 모드 활용
Claude Opus / Sonnet의 Thinking 모드는 복잡한 기획 결정이나 아키텍처 설계 시 활성화.  
단, **요청당 2배 비용** 소모되므로 꼭 필요할 때만 사용.

### Max Mode 전략
GPT-5.4 Max Mode는 **캐시 90% 할인 + 1M 컨텍스트**.  
대형 프로젝트 전체 분석 시 오히려 경제적일 수 있음.

### Agent 모드 활용
복잡한 코딩 작업은 `Agent` 기능이 있는 모델 선택.  
여러 파일 동시 수정, 터미널 명령 실행까지 자동화 가능.

### 기획 → 코딩 전환 워크플로우
기획서 작성은 **Claude Sonnet**, 구현은 **GPT Codex**로 전환하는 워크플로우가 효율적.

### 이미지 첨부 기획
UI 스크린샷, 와이어프레임 분석은 **Gemini 3.1 Pro** 또는 **Claude Sonnet** (Image 지원 모델) 사용.

### Auto 라우터 활용
Cursor의 **Auto 모드**는 작업 유형에 따라 비용·성능을 자동 최적화.  
처음 사용 시 Auto로 시작하고, 익숙해지면 수동 선택 권장.

## 8. Cursor에서 “좋은 결과”를 만드는 입력 방식 (실전)

Cursor에서 성능은 “모델 선택”만큼이나 **컨텍스트(어떤 파일/요구사항을 넣는지)** 와 **요청 형태(에이전트에게 시키는 방식)** 에서 갈립니다.

### 8.1 컨텍스트 넣는 3가지 레벨

- **레벨 1 — 대화로만 설명**  
  - 적합: 작은 질문, 짧은 코드 스니펫  
  - 한계: 실제 파일 구조/타입/의존성 반영이 약해짐

- **레벨 2 — 파일/폴더를 명시해서 참조**  
  - 적합: 실제 코드 변경, 리팩토링, 버그 수정  
  - 방법: 메시지에 `@파일경로` 또는 `@폴더경로`를 포함해 “근거”를 강제

- **레벨 3 — 변경 범위/금지사항/검증까지 포함한 작업 지시**  
  - 적합: 멀티파일 기능 개발, 안전한 리팩토링  
  - 핵심: “무엇을/어디에/어떤 기준으로/어떻게 검증할지”를 같이 줌

### 8.2 요청(프롬프트) 구성 템플릿

아래 순서로 쓰면, 같은 모델이라도 결과 품질이 크게 올라갑니다.

1) **목표(Outcome)**: 최종 결과물 형태(예: 컴포넌트 분리, 테스트 추가, 문서 갱신)  
2) **제약(Constraints)**: 금지사항/스타일/호환성(예: Node 버전, 타입스크립트 strict)  
3) **범위(Scope)**: 변경해도 되는/안 되는 경로, 파일 수, API 표면  
4) **성공 기준(Acceptance Criteria)**: “완료”를 판단할 체크리스트  
5) **검증(Test/Check)**: 실행 명령, 린트, 빌드, 스냅샷 등

## 9. Agent 모드 운영법 (멀티파일 작업을 “안전하게”)

Agent는 강력하지만, 잘못 쓰면 불필요한 파일 변경/과도한 수정이 발생합니다. 아래 패턴이 안전합니다.

### 9.1 권장 3-스텝 패턴

1) **탐색(읽기) 단계**  
   - “관련 파일을 찾아서 근거를 요약해줘”  
   - 목표: 어디를 고쳐야 하는지 정확히 좁히기

2) **설계(계획) 단계**  
   - “변경안 2~3개와 트레이드오프를 비교해줘”  
   - 목표: 접근법 확정(이 단계에서 모델을 Opus/Sonnet로 올리면 효율적)

3) **실행(수정) 단계**  
   - “이 계획대로 적용하고, 린트/테스트까지 돌려줘”  
   - 목표: 실제 코드 변경 + 검증 + 리그레션 방지

### 9.2 Agent에게 주면 좋은 “안전장치” 문장

- **변경 제한**: “`src/foo/` 아래만 변경해줘. 다른 파일은 건드리지 마.”  
- **작게 커밋**: “큰 리팩토링이면 2~3개의 작은 단계로 나눠서 진행해줘.”  
- **호환성**: “외부 API(함수 시그니처/URL)는 유지해줘.”  
- **검증**: “변경 후 빌드/린트/테스트를 실행하고 결과를 요약해줘.”

## 10. 작업 유형별 “바로 복붙” 프롬프트 예시

아래는 실제로 품질이 잘 나오는 지시 형태입니다. (파일 참조는 프로젝트에 맞게 `@`로 바꿔서 사용)

### 10.1 버그 원인 분석(재현 → 원인 → 수정 → 검증)

```
@src/ @package.json
증상: <무슨 에러/오동작>
재현: <재현 단계>
기대: <원래 기대 동작>

요청:
- 관련 코드 흐름을 추적해서 원인을 1~2개 후보로 좁혀줘
- 가장 가능성 높은 원인에 대해 최소 변경으로 수정해줘
- 변경 후 린트/테스트/빌드를 실행해 검증해줘
```

### 10.2 리팩토링(범위/성공기준을 명확히)

```
@src/components/ @src/utils/
목표: <예: 중복 로직 제거 + 타입 안정성 강화>
제약:
- 퍼블릭 API(컴포넌트 props, exported 함수 시그니처) 변경 금지
- 동작 변경 금지(동일 입력 → 동일 출력)
성공 기준:
- 중복 코드 제거(최소 N곳)
- 타입 에러 0
- 린트/테스트 통과
```

### 10.3 문서/가이드 작성(“대상 독자”를 고정)

```
대상 독자: <초급/중급/팀 내부>
목표: <예: 온보딩 30분 내 완료>
형식: 체크리스트 + 예시 + 흔한 실수 섹션 포함
범위: <예: 개발환경 세팅 ~ 첫 PR까지>
```

## 11. 비용/속도 실전 감각 (캐시/Max/Think를 언제 쓰나)

### 11.1 캐시(반복 입력)로 절약되는 경우

다음 상황은 같은 컨텍스트(긴 코드/문서)를 여러 번 넣게 되므로, **캐시 할인**이 큰 모델이 유리해집니다.

- 같은 폴더를 계속 참조하며 리팩토링 반복
- 큰 설계 문서를 기반으로 PRD/테스트/코드로 여러 번 전개
- 동일한 코드베이스를 “분석 → 계획 → 실행”으로 2~3회 왕복

### 11.2 Think(Thinking) 모드는 “계획” 단계에서만 켠다

- 켜면 좋은 때: 아키텍처 결정, 대규모 리팩토링 설계, 까다로운 버그 원인 후보 비교  
- 끄는 게 좋은 때: 단순 코드 생성/변경, 반복 작업(검색/치환/포맷팅 등)

### 11.3 Max Mode는 “한 번에 큰 그림”이 필요할 때만

- 적합: 레거시 마이그레이션, 모듈 경계 재설계, 전체 코드베이스 영향 분석  
- 비적합: 단일 기능 추가/작은 버그(오히려 느리고 비용 증가)

## 12. 자주 실패하는 패턴과 처방

### 12.1 “모델은 좋은데 결과가 엉뚱함”

- **원인**: 파일/폴더 컨텍스트 부족, 요구사항이 “목표”가 아니라 “방법”만 있음  
- **처방**: `@`로 실제 코드 근거를 붙이고, 성공 기준(체크리스트)을 추가

### 12.2 “과도하게 많은 파일을 바꿈”

- **원인**: 범위 제한이 없어서 Agent가 ‘최적화’하려고 확장  
- **처방**: 변경 가능 경로를 명시(예: `src/foo/`만), 금지 파일 지정

### 12.3 “코드가 돌아가긴 하는데 팀 스타일과 다름”

- **원인**: 린트/포맷/컨벤션이 지시되지 않음  
- **처방**: 린트/포맷 도구(ESLint/Prettier/ruff 등) 실행을 성공 기준에 포함

### 12.4 “설명은 맞는데 실제 코드베이스와 불일치”

- **원인**: 모델이 일반론으로 답함(실제 코드 읽기 부족)  
- **처방**: “추측 금지, 반드시 `@`로 준 파일을 근거로만 답해줘”를 추가
