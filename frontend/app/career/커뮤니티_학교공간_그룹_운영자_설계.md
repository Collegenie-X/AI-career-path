# DreamPath 커뮤니티 설계서
## 학교 공간 · 그룹 · 운영자 시스템 · 가입 프로필

> **핵심 원칙**: 학교 공간은 **승인 기반 인증 공간**(키즈 카페), 그룹은 **자유 운영 공간**.  
> 운영자 시스템으로 두 공간 모두 체계적 관리. 가입은 **유튜브처럼 최소 정보**(이름·학년)로 진입장벽 최소화.

---

## 1. 학교 공간 vs 그룹 차이점

| 구분 | 학교 공간 (School Space) | 그룹 (Group) |
|------|--------------------------|--------------|
| **성격** | 인증·승인 기반 공간 | 자유 생성·참여 공간 |
| **생성** | 플랫폼/교육청 승인 필요 | 누구나 생성 가능 |
| **비유** | 키즈 카페 (승인 후 운영) | 일반 카페/동호회 |
| **운영자** | 진로 교사·담당자 (승인된 1명) | 그룹 생성자 (멤버 중 지정) |
| **참여** | 학교 코드 입력 → 즉시 참여 | 초대 코드/링크로 참여 |
| **공유 범위** | 같은 학교 학생 + 운영자 | 그룹 멤버만 |
| **운영자 공유** | ✅ 지원 (선생님만 열람) | ✅ 지원 (그룹 운영자만 열람) |

### 1-1. 학교 공간 — 승인 기반 시스템 (키즈 카페 모델)

```
[학교 공간 생성 흐름]

1. 진로 교사/담당자 → "학교 공간 개설 신청"
2. DreamPath 플랫폼 → 신청서 검토 (학교 인증 서류 등)
3. 승인 → 학교 코드 발급 (예: FUTURE2024)
4. 교사 → 학생에게 코드 공유
5. 학생 → 코드 입력 → 학교 공간 참여
```

**학교 공간 상태**

| 상태 | 설명 |
|------|------|
| `pending` | 신청 접수, 검토 대기 |
| `approved` | 승인 완료, 운영 가능 |
| `suspended` | 운영 중단 (정책 위반 등) |
| `closed` | 폐쇄 |

### 1-2. 그룹 — 자유 운영 시스템

```
[그룹 생성 흐름]

1. 사용자 → "새 그룹 만들기"
2. 그룹명·설명·아이콘 입력
3. 즉시 생성 → 초대 코드/링크 발급
4. 친구 초대 → 참여
```

**그룹 상태**

| 상태 | 설명 |
|------|------|
| `active` | 정상 운영 |
| `archived` | 보관 (활동 중단) |
| `deleted` | 삭제됨 |

## 2. 운영자 시스템

### 2-1. 운영자 역할 정의

| 공간 유형 | 운영자 | 역할 |
|----------|--------|------|
| **학교 공간** | 진로 교사/담당자 | 패스 열람(운영자 공유), 피드백, 학교 공간 설정 |
| **그룹** | 그룹 생성자 또는 지정된 멤버 | 멤버 관리, 패스 열람(운영자 공유), 그룹 설정 |

### 2-2. 운영자 권한 매트릭스

| 권한 | 학교 공간 운영자 | 그룹 운영자 |
|------|------------------|-------------|
| 공유 패스 중 "운영자 공유" 열람 | ✅ | ✅ |
| 운영자 코멘트 작성 | ✅ | ✅ |
| 멤버 목록 조회 | ✅ | ✅ |
| 멤버 강제 퇴장 | ✅ (학생) | ✅ |
| 공간 설정 변경 | ✅ | ✅ |
| 공간 폐쇄/삭제 | ❌ (플랫폼만) | ✅ (생성자) |
| 학교 공간 신청/승인 | 신청 가능 | - |

### 2-3. 운영자 대시보드 (학교 공간)

```
[학교 공간 운영자 화면]

├── 개요
│   ├── 참여 학생 수
│   ├── 공유 패스 수 (전체/운영자 공유)
│   └── 학년별 분포
├── 공유 패스 관리
│   ├── 전체 공유 패스 목록
│   ├── 운영자 공유 패스만 보기 (피드백 대기)
│   └── 코멘트 작성/수정
├── 멤버 관리
│   ├── 참여 학생 목록
│   └── (필요 시) 퇴장 처리
└── 설정
    ├── 학교 정보 수정
    └── 학교 코드 재발급 (보안)
```

### 2-4. 운영자 대시보드 (그룹)

```
[그룹 운영자 화면]

├── 개요
│   ├── 멤버 수
│   └── 공유 패스 수
├── 멤버 관리
│   ├── 멤버 목록
│   ├── 운영자 지정/해제
│   └── 멤버 퇴장
├── 공유 패스
│   ├── 그룹 내 공유 패스 목록
│   └── 운영자 공유 패스 피드백
└── 설정
    ├── 그룹명·설명·아이콘 수정
    ├── 초대 코드 재발급
    └── 그룹 아카이브/삭제
```

## 4. 가입 프로필 — 최소 정보 (유튜브 스타일)

### 4-1. 설계 원칙

| 원칙 | 설명 |
|------|------|
| **진입장벽 최소화** | 가입 시 필수 입력 최소화 → 이탈 방지 |
| **학년별 맞춤** | 학년 정보로 추천·필터링·UI 맞춤 |
| **점진적 프로필** | 나중에 설정에서 추가 입력 유도 |

### 4-2. 가입 시 필수 입력 (최소)

| 필드 | 필수 | 설명 |
|------|------|------|
| **닉네임** | ✅ | 2~10자, 서비스 내 표시명 |
| **학년** | ✅ | 초4~고3 (GRADE_OPTIONS) |

### 4-3. 가입 시 선택 입력 (나중에 가능)

| 필드 | 시점 | 용도 |
|------|------|------|
| 프로필 이모지 | 설정 | 카드/댓글 표시 |
| 관심 직업 | 탐색 후 | 추천 개인화 |
| 학교 | 학교 공간 참여 시 | 학교 코드와 연동 |

### 4-4. 학년별 맞춤형 서비스 활용

| 학년 | 활용 예시 |
|------|-----------|
| 초4~6 | 초등용 직업 카드, 단순한 UI, 게임형 탐색 |
| 중1~3 | 중학교 로드맵, 동아리·자유학기 활동 추천 |
| 고1~3 | 고등 로드맵, 대입·생기부·포트폴리오 연계 |

### 4-5. UserProfile 스키마 (최소)

```typescript
interface UserProfile {
  id: string;
  nickname: string;        // 필수
  grade: string;           // 필수 (elem4 | mid1 | high2 등)
  emoji?: string;         // 선택, 기본값 랜덤
  joinedSchoolId?: string; // 학교 참여 시
  createdAt: string;
  onboardingCompleted?: boolean;
}
```

## 5. API 엔드포인트 (백엔드 연동 시)

### 5-1. 학교 공간

| Method | Endpoint | 설명 |
|--------|----------|------|
| POST | `/api/schools/applications` | 학교 공간 개설 신청 |
| GET | `/api/schools/applications/:id` | 신청 상태 조회 |
| GET | `/api/schools` | 승인된 학교 목록 (검색) |
| POST | `/api/schools/join` | 학교 코드로 참여 |
| GET | `/api/schools/:id` | 학교 상세 (운영자용) |
| PATCH | `/api/schools/:id` | 학교 정보 수정 (운영자) |

### 5-2. 그룹

| Method | Endpoint | 설명 |
|--------|----------|------|
| POST | `/api/groups` | 그룹 생성 |
| GET | `/api/groups/:id` | 그룹 상세 |
| PATCH | `/api/groups/:id` | 그룹 수정 (운영자) |
| POST | `/api/groups/join` | 초대 코드로 참여 |
| POST | `/api/groups/:id/members/:userId/role` | 운영자 지정 |
| DELETE | `/api/groups/:id/members/:userId` | 멤버 퇴장 |

### 5-3. 사용자/프로필

| Method | Endpoint | 설명 |
|--------|----------|------|
| POST | `/api/auth/signup` | 가입 (nickname, grade 필수) |
| PATCH | `/api/users/me` | 프로필 수정 (점진적) |
| GET | `/api/users/me` | 내 프로필 조회 |

## 6. UI/UX 가이드

### 6-1. 학교 공간 개설 신청 (교사용)

- 별도 "학교 공간 신청" 페이지/플로우
- 학교명, 학교급, 지역, 담당자 정보 입력
- 인증 서류 업로드 (선택, Phase 2)
- 신청 후 "검토 중" 상태 표시
- 승인 시 이메일/알림 + 학교 코드 전달

### 6-2. 그룹 생성 (학생용)

- 기존과 동일: 그룹명, 설명, 아이콘
- 생성 즉시 초대 링크/코드 발급
- 운영자 = 생성자 (기본)

### 6-3. 가입 플로우 (최소)

```
1. 스플래시
2. 온보딩 슬라이드 (앱 소개)
3. 프로필 설정
   - 닉네임 입력 (2~10자)
   - 학년 선택 (초4~고3 칩)
4. "모험 시작하기" → 메인
```

## 8. 구현 우선순위

| Phase | 항목 | 비고 |
|-------|------|------|
| **Phase 1** | 학교 공간 vs 그룹 개념 정리, 타입/스키마 확장 | 문서화 완료 |
| **Phase 1** | 가입 시 닉네임+학년만 (기존 Onboarding 유지) | 이미 적용됨 |
| **Phase 2** | 학교 공간 신청 플로우 (교사용) | 백엔드 필요 |
| **Phase 2** | 그룹 운영자 지정 UI | 프론트 확장 |
| **Phase 3** | 운영자 대시보드 (학교/그룹) | 백엔드 + 프론트 |
| **Phase 3** | 학년별 맞춤 UI/추천 | 점진적 적용 |

## 9. 요약

1. **학교 공간** = 승인 기반(키즈 카페). 신청 → 검토 → 승인 → 코드 발급.
2. **그룹** = 자유 생성. 즉시 생성, 초대 코드로 참여.
3. **운영자** = 학교(진로 교사), 그룹(생성자/지정 멤버). "운영자 공유" 패스 열람·피드백.
4. **가입** = 닉네임 + 학년만. 유튜브처럼 간단. 학년별 맞춤 서비스 가능.
